# מדריך משתמש עסקי

## מהו קישור תשלום

קישור תשלום הוא קישור מאובטח לדף תשלום מתארח של PayPlus. הלקוח פותח את הקישור, מזין את פרטי הכרטיס בדף PayPlus ומשלים את התשלום מחוץ ל-Dynamics 365 ומחוץ ל-Power Automate.

המערכת העסקית יכולה לשמור את בקשת התשלום, הסכום, הלקוח, הקישור, מזהי PayPlus והסטטוס. אין לשמור את מספר הכרטיס או CVV של הלקוח.

## מתי משתמשים בו

משתמשים בקישור תשלום כאשר לקוח צריך לשלם מרחוק והארגון רוצה לעקוב אחר הבקשה מתוך Dynamics 365 או תהליך Power Automate.

דוגמאות נפוצות:

- נציג שירות שולח בקשת תשלום לאחר טיפול בפנייה או הזמנה.
- צוות גבייה שולח קישור עבור יתרה פתוחה.
- צוות מכירות שולח קישור עבור הזמנה או מקדמה.
- צוות Back office יוצר קישור מתוך תהליך מאושר.

## איך נראה התהליך העסקי

1. המשתמש פותח רשומה ב-Dynamics 365 או מפעיל Flow מאושר.
2. המשתמש מאשר לקוח, סכום, מטבע, דף תשלום ותיאור.
3. המערכת יוצרת קישור תשלום ב-PayPlus.
4. הקישור נשמר ברשומת בקשת התשלום.
5. הקישור נשלח ללקוח בדוא"ל, SMS, צ'אט או ערוץ מאושר אחר.
6. הלקוח משלם בדף התשלום המתארח של PayPlus.
7. המערכת מעדכנת את הבקשה ידנית, באמצעות שליפת סטטוס, או באמצעות webhook אם יושם.

## מה המשתמש צריך למלא

הטופס המדויק תלוי ביישום, אך בדרך כלל המשתמש ממלא או מאשר:

| שדה | משמעות |
| --- | --- |
| לקוח | האדם או החשבון שאמור לשלם |
| סכום | הסכום לגבייה |
| מטבע | בדרך כלל ILS, אלא אם אושר מטבע אחר |
| דף תשלום | דף PayPlus המשמש לתרחיש התשלום |
| מסוף | מסוף PayPlus, אם השדה מוצג בטופס |
| תיאור או אסמכתה | סיבה עסקית, חשבונית, הזמנה או מספר פנייה |
| פרטי התקשרות | דוא"ל או טלפון אם הקישור נשלח אוטומטית |

## מה הלקוח מקבל

הלקוח מקבל קישור לדף תשלום של PayPlus. הדף מתארח אצל PayPlus ושם הלקוח מזין את פרטי הכרטיס.

יש לשלוח את הקישור רק בערוצי תקשורת מאושרים.

## מה קורה אחרי תשלום

בהתאם ליישום, המערכת יכולה לשמור:

- מזהה בקשת תשלום של PayPlus.
- קישור תשלום.
- מזהה עסקה.
- סטטוס תשלום.
- סכום ומטבע ששולמו.
- תאריך תשלום.
- אסמכתת אישור או שובר.
- ארבע ספרות אחרונות של כרטיס, אם PayPlus מחזיר אותן ואם אושר לשמור אותן.
- סיבת כשל או דחייה.

אין לשמור מספר כרטיס גולמי, CVV או פרטי כרטיס מלאים.

## סטטוסים צפויים

סטטוסים עסקיים מומלצים:

| סטטוס | משמעות |
| --- | --- |
| טיוטה | הבקשה הוכנה אך עדיין לא נוצר קישור |
| נוצר קישור | PayPlus החזיר קישור תשלום |
| נשלח | הקישור נשלח ללקוח |
| שולם | התשלום הושלם בהצלחה |
| נכשל | ניסיון התשלום נכשל או נדחה |
| פג תוקף | הקישור כבר אינו תקף |
| בוטל | הבקשה בוטלה על ידי הארגון |
| זוכה | התשלום זוכה באופן מלא או חלקי |
| דורש בדיקה | נדרשת בדיקה ידנית או התאמה |

## מה לעשות במקרה של כשל

- בדקו שפרטי הלקוח והסכום נכונים.
- בדקו שדף התשלום והמסוף נכונים.
- צרו קישור חדש רק אם המדיניות העסקית מאפשרת זאת.
- אל תבקשו מהלקוח לשלוח פרטי כרטיס בדוא"ל, צ'אט, תמלול שיחה או טקסט חופשי.
- העבירו לטיפול תמיכה או כספים אם הסטטוס אינו ברור.
- אם הלקוח טוען ששילם אך Dynamics לא עודכן, בקשו לבדוק את סטטוס העסקה ב-PayPlus או את לוג ה-webhook.

## מה לא לעשות

- לא להקליד מספרי כרטיס ב-Dynamics 365, הערות, שדות דוא"ל, קלטים של Flow או רשומות Dataverse.
- לא לשמור CVV בשום מקום.
- לא להדביק API keys או secret keys ברשומות, הערות, דוא"ל או קלטי פעולה.
- לא להשתמש באותו קישור עבור לקוח או סכום אחרים.
- לא להשתמש בקישורי ייצור לצורך בדיקות.
- לא לסמן בקשה כשולמה בלי אישור סטטוס תשלום.

## הסבר פשוט על מסוף ודף תשלום

מסוף הוא ההקשר המסחרי ב-PayPlus שבו התשלום מעובד. דף תשלום הוא הדף שהלקוח רואה ושייך למסוף מסוים.

בפשטות:

- מסוף: המקום שבו התשלום נסלק.
- דף תשלום: הדף שהלקוח פותח כדי לשלם.

בשלב ההגדרה מנהל עשוי לבחור מסוף ודף תשלום. משתמשים רגילים עשויים לראות רק שמות ידידותיים.

## למה לא מזינים כרטיס אשראי בתוך Dynamics או Flow

Dynamics 365 ו-Power Automate הן מערכות תהליך עסקי. בפתרון זה הן אינן מסכי הזנת כרטיס. הזנת מספרי כרטיס או CVV לתוכן תגדיל סיכוני אבטחה ו-PCI ועלולה לחשוף מידע בהיסטוריית ריצות, לוגים, יצוא או הערות.

PayPlus מספקת דף תשלום מתארח כדי שפרטי הכרטיס יישארו בצד PayPlus.
