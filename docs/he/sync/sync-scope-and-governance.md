# סנכרון רציף PayPlus - גבולות סנכרון ו-Governance

## עקרון גבול הסנכרון

לא מסנכרנים טבלת Dataverse רק כי המנוע הטכני יודע למפות אותה.

טבלה צריכה להיות מופעלת לסנכרון רציף רק כאשר כל התנאים מתקיימים:

- ל-PayPlus יש יעד אמיתי עבור המידע.
- אפשר ליצור או לעדכן את היעד באופן אוטומטי ובטוח.
- בעל התהליך העסקי יודע להסביר למה PayPlus צריך את המידע.
- מקור האמת ברור.
- כללי הזכאות ברורים.
- אסטרטגיית המזהים ברורה.
- ההשפעה של כפילויות, עדכונים חסרים והשבתה מובנת.

אם הערך העסקי אינו ברור, התשובה הנכונה היא לעצור ולשאול את בעל התהליך. אינטגרציה גנרית לא צריכה להמציא תהליך עסקי.

## יעדי סנכרון רציף מומלצים

### לקוחות

מקורות מומלצים:

- `account` עבור לקוחות עסקיים.
- `contact` עבור לקוחות פרטיים או אנשי קשר שמשלמים.

יעד PayPlus מומלץ:

- Customer.

למה מומלץ:

- רשומות לקוח ב-PayPlus שימושיות לקישורי תשלום, כרטיסים שמורים, איתור לקוח, תשלומים חוזרים ומסמכים.
- סמנטיקת Create/Update יחסית פשוטה.
- ניתן לשמור `customer_uid` של PayPlus חזרה ב-Sync State.

כללי זהירות חשובים:

- לא למפות GUID של Dataverse לשדות PayPlus שמצפים לקוד עסקי קצר.
- `subject_code` לא צריך לקבל GUID של Dataverse אלא אם PayPlus תומך בכך במפורש עבור אותו לקוח. במימוש הנוכחי לרוב יש להשאיר אותו לא ממופה.
- להשתמש בכללי סינון כדי לסנכרן רק לקוחות שרלוונטיים לפעילות תשלום.
- להשתמש במזהים עסקיים כגון מספר לקוח, ח.פ/ת.ז או אימייל רק כאשר הם מנוהלים ויציבים.

### מוצרים

מקור מומלץ:

- `productpricelevel` עם `product` ושדות מטבע קשורים.

יעד PayPlus מומלץ:

- Product.

למה מומלץ:

- קטלוג PayPlus צריך מחיר מכירה, מטבע, התנהגות מע"מ, שם מוצר וברקוד/SKU.
- מחיר מוצר ב-Dynamics בדרך כלל תלוי מחירון, ולכן `productpricelevel` הוא מקור נכון יותר למחירי PayPlus מאשר `product` בלבד.

תפיסת מיפוי מומלצת:

| שדה PayPlus | ערך Dynamics מומלץ |
| --- | --- |
| `name` | `product.name` |
| `price` | `amount` |
| `currency_code` | קוד ISO של מטבע קשור |
| `barcode` | `product.productnumber` / SKU |
| `description` | `product.name` או תיאור מבוקר |
| `valid` | קבוע `true` או המרה מסטטוס |
| `vat_type` | קבוע מבוקר או מיפוי מדיניות מס |
| `category_uids` | רשימת UID של קטגוריות PayPlus מבוקרת |

כללי זהירות חשובים:

- לא לשלוח GUID של מוצר Dynamics כמזהה מוצר PayPlus.
- `uid` / `product_uid` של PayPlus נוצר על ידי PayPlus ונשמר ב-Sync State.
- `barcode` הוא השדה העסקי המעשי עבור SKU / מספר מוצר.
- אם Sync State חסר, אפשר לבצע התאוששות באמצעות Lookup לפי `barcode` ומטבע לפני יצירת כפילות.

### קטגוריות מוצרים

מקור מומלץ:

- טבלת Reference מבוקרת ב-Dataverse או קונפיגורציה מנוהלת.

יעד PayPlus מומלץ:

- Product Category.

למה מומלץ:

- מוצרים צריכים UID של קטגוריות ב-PayPlus.
- קטגוריות הן Reference Data יציב.
- סמנטיקת Create/Update לרוב פשוטה.

כללי זהירות חשובים:

- לא לגזור קטגוריות PayPlus משדות יחידות מידה או היררכיית מוצר שאינם מייצגים קטגוריית PayPlus אלא אם העסק אישר זאת במפורש.
- לשמור מזהי קטגוריה יציבים.
- לא לשנות מבנה קטגוריות אוטומטית מתוך נתונים תפעוליים תנודתיים.

## יעדים מותנים: להשתמש ב-Flow, לא בסנכרון רציף גנרי

היעדים הבאים יכולים להיות חלק מהאינטגרציה, אך בדרך כלל צריכים להתממש כתהליכים עסקיים ייעודיים ולא כסנכרון טבלאות קבוע.

| יעד | דפוס מומלץ | למה לא סנכרון רציף גנרי |
| --- | --- | --- |
| קישור תשלום / דרישת תשלום | Flow ייעודי שמופעל מפעולת משתמש או אבן דרך עסקית. | יוצר פעולה מול לקוח; לא אמור לקרות בכל עדכון רשומה. |
| הצעת מחיר / פרופורמה / הזמנה / מסמך רכש | Flow הפקת מסמך מבוקר. | להפקת מסמך יש מספור, משפטיות, מחזור חיים וסיכון כפילויות. |
| חשבונית / קבלה / זיכוי | Flow מאושר פיננסית ומפורש. | מסמכי מס חייבים לצאת מאירוע עסקי מכוון. |
| תשלום חוזר | Flow Onboarding או חוזה מאושר. | דורש הרשאה, תאריך התחלה, לוח חיובים, הסכמת לקוח וטיפול בכשל. |
| חיוב כרטיס שמור | Flow תשלום ייעודי עם טוקן PayPlus שמור. | מדובר בחיוב כספי ולכן חייב Governance, Audit ואישור. |
| סטטוס עסקה | Polling או Reconciliation מ-PayPlus ל-Dataverse. | PayPlus הוא מקור האמת לסטטוס עסקה; זה לא Sync Master Data מ-Dataverse ל-PayPlus. |
| כרטיסים מטוקנים | Hosted Fields / Self-service session flows. | טוקנים רגישים ונוצרים בתהליכי טוקניזציה של PayPlus, לא בסנכרון טבלה גנרי. |

## טבלאות שלא מומלץ לסנכרן גנרית

### טבלאות Pipeline מכירות ושירות

דוגמאות:

- Leads.
- Opportunities.
- Cases.
- Activities.
- Tasks, Phone Calls, Appointments, Emails.

למה לא מומלץ:

- אין להן מקבילה ישירה כ-Master Data ב-PayPlus.
- מחזור החיים שלהן הוא תהליך מכירות/שירות, לא ביצוע תשלום.
- סנכרון כל עדכון יוצר רעש, Side Effects ועומס תמיכה.
- הערך העסקי בדרך כלל לא ברור אלא אם יש תהליך תשלום ספציפי שמחובר אליהן.

חלופה מומלצת:

- להפעיל קישור תשלום או הפקת מסמך רק בשלב מוגדר, כגון הצעת מחיר מאושרת, הזמנה מאושרת או דרישת גבייה.

### טבלאות פיננסיות ומסמכים

דוגמאות:

- Quotes.
- Orders.
- Invoices.
- Invoice Lines.
- Payments.
- Credit Memos.

למה לא מומלץ כסנכרון רציף גנרי:

- API המסמכים של PayPlus מפיק תוצרים תפעוליים או משפטיים.
- עדכונים חוזרים עלולים ליצור כפילויות או גרסאות מסמך לא עקביות.
- סיכומים, מע"מ, הנחות, עיגול, מטבע, זהות לקוח ושורות חייבים להיבדק כעסקה עסקית אחת.
- מסמכים משפטיים דורשים לרוב אישור מפורש, מספור וכללי ביטול.

חלופה מומלצת:

- להשתמש ב-Flow ייעודי להפקת מסמך שבונה Payload מלא, בודק Idempotency באמצעות מזהה ייחודי, ושומר UID של מסמך PayPlus.

### טבלאות מערכת, אבטחה ופלטפורמה

דוגמאות:

- Users.
- Teams.
- Business Units.
- Roles.
- Audit.
- Plugin Trace Logs.
- Flow Run Logs.

למה לא מומלץ:

- ל-PayPlus אין יעד עסקי לטבלאות אלו.
- הן מכילות מידע מנהלי או אבטחתי.
- סנכרון שלהן לא מוסיף ערך תשלום ועלול ליצור סיכון Compliance.

חלופה מומלצת:

- להשאיר את הטבלאות האלו פנימיות ל-Dataverse ולניטור Power Platform.

### טבלאות Marketing, Consent והעדפות תקשורת

דוגמאות:

- רשומות Consent.
- Marketing Lists.
- Contact-point Consent.
- Campaign Interactions.

למה לא מומלץ:

- משמעות Consent רגישה לפרטיות ותלויה ב-Tenant ובמדיניות.
- שדות לקוח/איש קשר ב-PayPlus אינם מחליפים מערכת Consent Management.
- שכפול לא נכון עלול להתנגש במדיניות פרטיות או בסיס חוקי.

חלופה מומלצת:

- להשתמש ב-Consent כתנאי זכאות לפני שליחת קישור תשלום או הודעת שירות עצמי.

### טבלאות Custom ללא יעד PayPlus מוגדר

למה לא מומלץ:

- טבלת Custom יכולה להיות ניתנת למיפוי טכנית, אבל בלי יעד PayPlus אין חוזה API יציב.
- מיפוי Payload גנרי לא יכול להחליט עיתוי עסקי, מניעת כפילויות או משמעות פיננסית.

חלופה מומלצת:

- קודם להגדיר את התהליך העסקי ואת יעד PayPlus. רק אחר כך להחליט אם המימוש הנכון הוא סנכרון רציף, Flow ייעודי או ללא אינטגרציה.

## Checklist להחלטה

לפני הפעלת Entity Mapping חדש יש להשתמש ב-Checklist הבא.

| שאלה | תשובה נדרשת |
| --- | --- |
| איזה יעד PayPlus יקבל את המידע? | יעד/פעולת PayPlus בשם ברור. |
| למה PayPlus צריך את המידע? | סיבה עסקית שקשורה לתשלום, מסמך, קטלוג או לקוח. |
| מי מחזיק את מקור הנתונים? | טבלת Dynamics וצוות עסקי אחראי. |
| מה מזהה Create/Update? | PayPlus UID ב-Sync State ובמידת הצורך מפתח עסקי יציב. |
| מה כלל הסינון? | תנאי זכאות מדויקים. |
| האם עדכון אוטומטי בטוח? | כן, או להשתמש ב-Flow אישור ייעודי במקום. |
| מה קורה כאשר המקור הופך ללא פעיל? | מדיניות Deactivate/Update ברורה. |
| איך מונעים כפילויות? | Sync State, מפתח ייחודי, Lookup-before-create או שדה Idempotency. |
| איך מנטרים כשלים? | Outbox Status, Sync Log ובעלים תפעולי. |

## כלל Governance מומלץ

סנכרון רציף צריך להיתפס כפרסום Master Data, לא כקיצור דרך אוטומציה כללי.

אם טבלה מייצגת Master/Reference Data ש-PayPlus צריך שוב ושוב, היא מועמדת לסנכרון רציף. אם טבלה מייצגת אירוע, אישור, פעולת תשלום, הפקת מסמך או שינוי מצב משפטי/פיננסי, יש להשתמש בתהליך Power Automate ייעודי עם בקרות מפורשות.
