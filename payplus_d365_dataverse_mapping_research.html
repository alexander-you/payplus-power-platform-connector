תקציר מנהלים

ההמלצה המרכזית היא להשתמש ב-Dynamics 365 Sales וב-Dataverse כמקור האמת ללקוחות, מוצרים, מחירונים, הצעות מחיר, הזמנות וחשבוניות מכירה, ולהשתמש ב-PayPlus כמקור האמת לתשלומים, טוקנים, Payment Pages, סטטוסי סליקה ומסמכי Invoice+ שנוצרים בפועל במערכת PayPlus. כך מנצלים את המודל הסטנדרטי של Dynamics, מצמצמים Custom Fields, ומוסיפים רק שכבת זיהוי, לוגים וקורלציה.

התאמה גבוהה OOTBCustomer, Product, Items, Amounts, Currency
פערים עיקרייםPayPlus UIDs, Tokens, Payment Link, Document UUID
מודל אינטגרציה מומלץDataverse + Power Automate + Custom Connector + Webhooks
החלטת SoR מומלצת: Dynamics מנהל את תהליך המכירה והמסמך העסקי; PayPlus מנהל את התשלום והמסמך הפיסקלי אם Invoice+ פעיל. ב-Dataverse נשמרים מזהי PayPlus, סטטוס מסונכרן, קישורים למסמכים ולוג אירועים, אך לא נשמרים מספרי כרטיס מלאים או CVV.

Currency Management

כיצד Dynamics שומר מטבע ושער

Dataverse הוא Multi-currency. כל רשומה פיננסית יכולה לשאת transactioncurrencyid. שער המטבע נשמר ב-TransactionCurrency.exchangerate, וברשומות Sales נחשף גם exchangerate read-only. לכל Money field נוצר אוטומטית שדה Base read-only כמו totalamount_base. לפי Microsoft, ערך הבסיס מחושב בנוסחה money_base = money / exchangerate. מקור: Transaction Currency table.

כיצד PayPlus מטפל במטבעות

PayPlus מקבל currency_code בבקשות Payment, Payment Page, Products, Recurring ו-Documents, ומספק endpoint של רשימת מטבעות נתמכים. ב-Invoice+ קיימים conversion_rate ו-autocalculate_rate למסמך במטבע שאינו בסיס. מקור: Currencies, Create new document.

Integration Design ו-Power Platform Best Practices

Dataverse

Use OOTB tables for business records. Add narrow custom tables for PayPlus transaction, document, payment request, token and mapping. Define alternate keys on PayPlus UIDs and use lookup relationships to Account, Contact, Product, Quote, Order and Invoice.

Power Automate

Use solution-aware cloud flows with connection references. Trigger from command bar buttons or Dataverse row changes: generate payment link, create/update PayPlus customer, charge token, refund, reconcile transaction, fetch PayPlus documents.

Custom Connector

Use one connector per PayPlus environment or a connector with environment-specific host. Store api-key and secret-key as secure connection parameters and inject headers with connector policies. This avoids passing keys per action.

Webhooks / Callbacks

Expose a Power Automate HTTP trigger, Azure Function or custom API endpoint for IPN/callbacks. Validate PayPlus callback signatures according to Validate Requests Received from PayPlus, then upsert transaction/document logs idempotently.

Environment Variables

Use environment variables for non-secret configuration: PayPlus environment, default payment page UID, callback URLs, feature flags, default terminal/cashier. Use connection parameters for the two PayPlus API secrets.

Azure Key Vault

Not required if Power Platform connector connection parameters are approved as the secret store. Use Azure Key Vault only if enterprise policy mandates centralized secret lifecycle, or if a server-side proxy/Azure Function is introduced.

המלצות ארכיטקטורה ויישום

  1. Use OOTB first: Account, Contact, CustomerAddress, Product, PriceLevel, ProductPriceLevel, Quote/Order/Invoice and line tables cover the majority of PayPlus fields.
  2. Use custom only for PayPlus identity and lifecycle: PayPlus UIDs, payment links, tokens, transaction logs, document logs and callback audit require custom storage.
  3. Hosted payment first: Prefer PaymentPages/generateLink for low-code scenarios. Direct charge should be token-only unless PCI scope is approved.
  4. Idempotency: Use Dynamics GUIDs as PayPlus unique_identifier for Invoice+ documents and store PayPlus response IDs as alternate keys.
  5. Callbacks are authoritative: Do not mark invoice/order paid only because a link was generated. Update payment status only after PayPlus transaction success callback or transaction lookup.
  6. Separate commercial and fiscal documents: Dynamics Invoice represents Sales billing context; PayPlus Invoice+ document represents fiscal/tax document. Link them, do not conflate them.
  7. ALM: Package connector, flows, connection references, custom tables and environment variables in a Dataverse solution. Use separate connection references for prod/sandbox.

מסקנות

האינטגרציה המומלצת היא Connector-centric על גבי Dataverse ו-Power Automate: Dynamics 365 Sales נשאר מקור האמת ל-Sales ולקטלוג, PayPlus נשאר מקור האמת לתשלום, טוקנים ומסמכי Invoice+, ושכבת Dataverse custom קטנה מספקת קורלציה, audit ו-reconciliation. זהו המודל שממזער Customizations: רוב השדות העסקיים ממופים ל-OOTB, וכל שדה Custom משרת צורך אינטגרטיבי ברור.

הפערים המשמעותיים אינם במודל המכירה אלא בזהויות התפעוליות של PayPlus: customer_uid, transaction_uid, page_request_uid, document_uuid, token_uid, terminal_uid/cashier_uid וסוגי מסמך/תשלום. לכן אין הצדקה לשכפל את כל מודל PayPlus ב-Dynamics; יש לבנות שכבת mapping/log צרה, עם alternate keys ותהליכי idempotent upsert.

המלצה סופית: להתחיל ב-slice מלא אחד: Account/Contact sync → GeneratePaymentLink מתוך Invoice/Order → Callback/IPN → Transaction Log → עדכון Invoice כ-Paid/Partially Paid → שליפת/שמירת PayPlus Document UUID. לאחר מכן להוסיף Token charge, Refund, Recurring ו-Product catalog sync לפי צורך עסקי.
