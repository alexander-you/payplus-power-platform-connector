תפיסת פתרון

הפתרון מבוסס על הפרדה בין הגדרה, זיהוי אירוע וביצוע סנכרון. Dynamics 365 הוא מקור האמת. PayPlus מקבל יצירה ועדכון בלבד. ה-Plugin לא קורא ל-PayPlus, אלא יוצר אירוע סנכרון בטבלת Outbox. Power Automate קורא את האירוע, בונה payload לפי מיפוי דינמי, מפעיל את ה-Custom Connector ומעדכן סטטוס.

עקרון ראשון Metadata-driven mapping

טבלאות ושדות אינם hardcoded בתוך Flow. המיפוי נשמר בטבלאות Dataverse כדי לתמוך גם ב-OOTB וגם בטבלאות לקוח.

עקרון שני Current state sync

בעדכונים נשלח המצב הנוכחי של הרשומה ב-D365, ולא בהכרח כל מצב ביניים שנוצר בזמן עריכה מקבילית.

עקרון שלישי Operational visibility

כל אירוע סנכרון מקבל סטטוס, מספר ניסיונות, שגיאה אחרונה ויכולת הרצה מחדש.

ארכיטקטורה תפעולית

הארכיטקטורה המומלצת ל-Production ללא Azure Function היא שילוב של Async Plugin, טבלת Outbox ו-Power Automate כמעבד הסנכרון.

1Create / Update

משתמש או תהליך מעדכן רשומה ב-Dynamics 365.

2Async Plugin

בודק אם הטבלה והשדות רלוונטיים לסנכרון לפי הגדרות.

3Outbox

נוצרת או מתעדכנת משימת סנכרון עם סטטוס Pending.

4Power Automate

קורא את הרשומה העדכנית, מפעיל מיפוי ושולח ל-PayPlus.

5State / Log

נשמר PayPlus UID, סטטוס, payload, response ושגיאות.

החלטת תכנון מרכזית

טבלת Outbox היא טבלת Dataverse רגילה ולא Elastic. היא תור עבודה קריטי עם סטטוס, retry וקשרים למיפוי ול-Sync State. אם יידרש ארכיון בנפח גבוה, ניתן לשקול Elastic רק עבור לוג היסטורי משני.

קשר דינמי לרשומות מקור

אין ליצור Lookup קשיח מ-Outbox או Sync State אל contact, product או טבלת לקוח מותאמת. במקום זאת נשמרים alex_sourcetablelogicalname ו-alex_sourcerowid. כך אותו מודל תומך גם ב-OOTB וגם בטבלאות מותאמות.

תהליכי יצירה ועדכון

Create יצירת Contact כ-Customer

בעת יצירת איש קשר שעומד בתנאי הסינון, ה-Plugin יוצר Outbox עם alex_operation = Create. ה-Flow קורא את ה-Contact, בונה payload לפי Field Mapping, מפעיל CreateCustomer ושומר PayPlus UID ב-Sync State.

Update עדכון מצב נוכחי

בעת עדכון, ה-Plugin משווה שדות שהשתנו מול Field Mapping. אם השתנה שדה ממופה, נוצרת או מתעדכנת משימת Outbox. ה-Flow קורא את הרשומה העדכנית ושולח את הערך האחרון.

Custom Table טבלת מוצרים מותאמת

אם לקוח בוחר טבלה כמו new_customerproduct, מוסיפים Entity Mapping ו-Field Mapping. ניתן לרשום Plugin Step לטבלה זו או להשתמש ב-polling כחלופה.

המלצה לפתיחה

להתחיל עם מיפוי אחד מלא: contact אל PayPlus Customer. זה מוכיח את כל רכיבי הארכיטקטורה בלי להיכנס עדיין למורכבות של מוצרים, מטבעות, קטגוריות ומסמכים פיננסיים.

מסמך תפיסת פתרון זה מתאר מודל מוצע. שמות הסכמה כוללים את קידומת ה-publisher alex ומיועדים ליישום כטבלאות Dataverse רגילות במסגרת פתרון D365 PayPlus.
